1. How the feed decides what to show you

The Home Mixer README describes the broad flow as candidate generation, feature hydration, scoring and ranking, filters and heuristics, then mixing. On this page, those are translated as shortlist building, adding context, scoring, filtering, and final assembly. The current recommendation pipeline in code provides the more exact list of active tweet sources.

Important accuracy note: the README's simplified For You diagram still mentions an FRS tweet pipeline, but the current ScoredTweetsRecommendationPipelineConfig candidate list does not include it. On this page, the current code wins when prose and code differ.
Plain-language guide: on this page, a candidate means a post that made it onto the system's shortlist; in-network means from people you follow; out-of-network means from outside your follows; UTEG is the repo's "liked by people near your graph" recommendation path; and Tweet Mixer is another service that supplies many interest-based recommendation pools.
Stage 1

Shortlist

Pull possible posts from follows, network-driven recommendations, interest systems, communities, and support paths.

Stage 2

Score

Add a large feature set and compute model predictions used for ranking.

Stage 3

Filter

Remove items that are too old, already seen, repetitive, invalid, or filtered for quality or safety reasons.

Stage 4

Assemble

Insert the remaining tweets into the timeline and then add non-tweet modules such as ads or who-to-follow.

First, the system builds a shortlist of possible posts.

Think of this as building a shortlist. The code does not start from every post on the platform and rank all of them. It first narrows the universe down to a much smaller set of candidates that look plausible for this viewer.

The active scored-tweets pipeline currently draws from static support paths, cached posts, posts from people you follow, the direct UTEG path, Tweet Mixer, content exploration, lists, backfill, and communities.

Interpretation

The feed starts with several shortlists for this viewer, not one giant ranking pass over every post on the platform.

2. What the feed prioritizes and how it reshapes rank

The code does not optimize for one action only. It predicts a set of outcome types, combines them into a final score, and then applies additional multipliers, caps, and selectors.

This visual lists prediction or filter types that explicitly exist in code. It does not show final numeric model-head weights, because those are not hard-coded here as a single stable set of constants.

The model predicts more than one kind of "good outcome."

The code estimates several outcome types at once. The same post can rank because it is predicted to cause a reply, or because it is predicted to cause a bookmark or longer dwell.

What the code clearly says

Prediction heads include favorite, reply, retweet, bookmark, share, good click, profile click, dwell, multiple video quality heads, watch time, and negative feedback.

Interpretation

The feed is not optimizing for one reaction only. It is combining several predicted outcomes.

4. Creator guidance derived from current code

The repo does not publish a creator handbook. So every recommendation below is labeled as an inference from implementation, not a guaranteed rule.

Inference from implementation: the code is more directly aligned with content that can be matched to an audience, can produce several positive outcome types, and does not trigger negative feedback or filters.
Inference from implementation

Do not rely on likes alone

The model predicts many outcomes: likes, replies, retweets, bookmarks, shares, good clicks, profile clicks, dwell, multiple video quality outcomes, watch time, and negative feedback.

Interpretation: the modeled objectives are broader than likes alone. The repo does not expose one published weight table, but it does clearly show multiple ranking heads.

Repo evidence
  • .../PredictedScoreFeature.scala, lines 62-311: the ranking heads that explicitly exist.
Inference from implementation

Make the topic easy to match

The retrieval stack includes category systems, communities, embeddings, topic and geo systems, semantic features, and media-based features.

Interpretation: clearer subject matter and audience fit should give the retrieval stack more ways to match a post to viewers.

Repo evidence
  • tweet-mixer/.../HomeRecommendedTweetsRecommendationPipelineConfig.scala, lines 271-332: broad set of interest-matching retrieval paths.
  • .../ScoredTweetsModelScoringPipelineConfig.scala, lines 203-306: semantic, topic, embedding, and multimodal features.
Inference from implementation

Replies face extra structural filters

The Home pipeline has explicit filters for some reply shapes, especially recommended replies to people the viewer does not follow, or reply structures that do not meet specific conditions.

Interpretation: replies can spread, but some reply shapes face extra gates that standalone posts do not.

Repo evidence
  • .../OONReplyFilter.scala, lines 16-40: some out-of-network replies are removed.
  • .../QualifiedRepliesFilter.scala, lines 28-42: some in-network reply structures are removed.
Inference from implementation

Video is judged on more than starts

The model includes video quality view, video quality watch, and watch-time heads. There is also a minimum video-duration filter.

Interpretation: video ranking includes attention-quality and watch-time style objectives, not just whether playback starts.

Repo evidence
  • .../PredictedScoreFeature.scala, lines 131-190 and 249-290: video-related ranking heads.
  • .../MinVideoDurationFilter.scala, lines 17-60: minimum duration filtering logic.
Inference from implementation

Negative feedback and filters matter

The system explicitly models negative feedback and filters for slop, spam, gore, NSFW, violent content, duplicate conversation items, and fatigue from "See fewer."

Interpretation: distribution depends on both positive engagement and on avoiding negative-feedback or filter-triggering conditions.

Repo evidence
  • .../PredictedScoreFeature.scala, lines 260-311: negative feedback head.
  • .../ScoredTweetsRecommendationPipelineConfig.scala, lines 532-556: post-scoring filters.
  • .../FeedbackFatigueFilter.scala, lines 39-79: fatigue logic.
Inference from implementation

Posting more often does not create equal extra reach

The ranking heuristics explicitly decay repeated posts from the same author and from the same source or reason. Seen or served items are also filtered out.

Interpretation: extra posting can help only if the posts do not crowd each other out in the same viewer's recommendation set.

Repo evidence

  • .../CandidateSourceDiversityListwiseRescoringProvider.scala, lines 26-49: source diversity decay.
  • .../PreviouslySeenTweetsFilter.scala, lines 32-45: already seen items are removed.
