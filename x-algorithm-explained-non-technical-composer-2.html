1. What you see in “For You” (according to home-mixer/README.md)

The docs in this repo say that Home Mixer builds the home timelines, and that For You is described as the “best” posts from people you follow plus recommended out-of-network content.

For the For You flow, the same README lists stages in order: gather candidate posts from several sources, load about 6000 features for ranking, run a machine learning model to score and rank, then apply filters and rules (author diversity, in-network vs out-of-network balance, feedback fatigue, deduplication, visibility rules), then mix in non-post items such as ads and suggestions.

Source: home-mixer/README.md (Overview and For You pipeline list).

2. What “Tweepcred” / author reputation is in this codebase

The root README.md describes tweepcred as a PageRank-style computation for user reputation. The Scala batch jobs under src/scala/com/twitter/graph/batch/job/tweepcred/ implement that story in three plain ideas:

  1. Build a graph of users (follow edges from Flock; optional weighted edges from “Real Graph” when configured).
  2. Run PageRank-style iterations with a random jump term; the default jump probability in code is 0.1 (10%).
  3. Convert the resulting score into a small integer (roughly 0–100) for storage and later use.

Sources: PreparePageRankData.scala, WeightedPageRank.scala, Reputation.scala, ExtractTweepcred.scala.

Code comment in WeightedPageRank.scala describes spreading mass along outgoing edges and mixing in a random jump. Flock snapshot age is configured as “no older than 7 days” in PreparePageRankData.scala for the read shown in this repo.

“User mass” before PageRank (from UserMass.scala)

Each user gets a starting weight used in the random-jump part of PageRank. The file encodes explicit branches, including:

Condition in codeEffect on mass in that branch
Account suspendedmass = 0
Account verified (safety flag in this struct)mass = 100
OtherwiseA formula using account age and device flags, then scaled to 0–100 range in that branch; restricted accounts get an extra multiplicative factor of 0.1

If someone follows more than 500 accounts and the ratio (1 + followings) / (1 + followers) is above 0.6, the code divides mass by an exponential penalty. Exact constants: threshAbsNumFriendsUMass = 500, threshFriendsToFollowersRatioUMass = 0.6, constantDivisionFactorGt_threshFriendsToFollowersRatioUMass = 5.0.

Source: UserMass.scala.

After PageRank: map to 0–100 and optional “follow-churn” adjustment

Reputation.scaledReputation maps the raw PageRank value with a natural logarithm and a linear formula, and clamps to a byte. The comments in that file say the fit maps roughly min PageRank → 15 and max PageRank → 95 on that scale (those two numbers appear in comments, not as runtime asserts).

ExtractTweepcred can apply adjustReputationsPostCalculation, which only does extra division when numFollowings > 2500, using another exponential based on followings-to-followers ratio (threshold threshFriendsToFollowersRatioUMass = 0.6 again in that function). Then it normalizes masses before writing output.

Sources: Reputation.scala, ExtractTweepcred.scala.

3. How reputation shows up when posts are indexed (Earlybird ingester)

UserPropertiesManager.java loads a column called TWEEPCRED from metastore and, when valid, sets userReputation on the message. If there is no usable value, it leaves the sentinel UNSET_REPUTATION_SENTINEL, which is Byte.MIN_VALUE (−128). RelevanceSignalConstants.java documents a goodwill replacement value of 17 for some scoring paths when reputation is unset (see comments in that file and usage in scoring code).

Sources: UserPropertiesManager.java, RelevanceSignalConstants.java, FeatureBasedScoringFunction.java (comments about skipping low reputation vs. unset).
